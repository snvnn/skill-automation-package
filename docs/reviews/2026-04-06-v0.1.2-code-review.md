## Findings

### [medium] Dry-run messaging is still only partially truthful

Why it matters:

- `--dry-run` now correctly avoids writing the install manifest and target directory.
- However, the CLI still prints `Updated AGENTS.md: yes` and `Updated CLAUDE.md: yes` when a dry-run detects that those files would change.
- That is easy to misread as a completed write rather than a preview result.

When it shows up:

- Any operator running `python3 scripts/install.py --target <repo> --dry-run` against a repo where the managed blocks would change.

Relevant references:

- `scripts/install.py:51`
- `scripts/install.py:60`
- `scripts/install.py:83`
- `scripts/install.py:125`
- `README.md:139`
- `tests/test_install.py:65`

### [low] Recommended upgrade flow is easy to skim incorrectly

Why it matters:

- The README's upgrade section shows a second command that uses `--skip-agents --skip-claude`.
- The limitation for that command is explained only after the command block.
- A skimming operator could treat that block as the default upgrade path and unintentionally leave managed guidance stale.

When it shows up:

- Existing users upgrading by scanning commands rather than reading the surrounding prose in detail.

Relevant references:

- `README.md:146`
- `README.md:173`
- `README.md:180`
- `scripts/install.py:51`

## Residual Risks / Testing Gaps

- No test currently locks the dry-run status lines for `AGENTS.md` and `CLAUDE.md`, which is why the remaining misleading output survived while the manifest line was fixed.
- The upgrade path is still not tested against a realistic pre-existing installation with existing `AGENTS.md`, existing `CLAUDE.md`, existing `usage.json`, and stale orphaned package files.
- `sync_assets.py` is tested for explicit roots and ancestor discovery, but not for ambiguous multi-match layouts or a colocated source-root layout.
- The README now recommends package.json maintenance scripts for bytecode-clean verification, but the raw unittest commands in the docs still do not themselves guarantee that behavior.

- Decide whether dry-run output should use `Would update ...` style wording instead of `Updated ...`.
- Add an integration-style reinstall test for an already-installed target repo.

## Summary

Most other concerns are documentation clarity and missing integration coverage around upgrade behavior.
